# GitHub Watcher カテゴリ分類・体系見直し提案

## 対象と前提

- 対象: `data/articles.json` 内の `github-trending` / `github-pickup`
- 分析対象記事数: 230
- 今回は**変更提案のみ**であり、実データや UI のカテゴリ変更は行わない

## 結論

現状の GitHub Watcher は、**カテゴリの大枠自体はおおむね有効**です。  
問題の中心は「カテゴリ数が足りないこと」ではなく、以下の 3 点です。

1. **ラベル揺れ**が残っている
2. **AI エージェント系のカテゴリが分裂**している
3. `汎用アプリ・AI基盤・IoT` のように**意味が広すぎるカテゴリ名**があり、割り当て判断がぶれやすい

したがって、全面的な再設計よりも、**既存カテゴリの統合・命名整理・運用ルール明確化**を優先するのが妥当です。

## 現状で妥当な点

以下は、現状のカテゴリを大きく崩さず維持してよい領域です。

- **金融・トレード分析**  
  役割が明確で、投資・分析系 repo を探したい利用者にとって価値が高い
- **ナレッジ管理・RAG解析**  
  利用シーンが比較的一貫しており、検索・RAG・知識統合の文脈でまとまっている
- **学習ガイド・開発アセット**  
  テンプレート、教材、実践ガイドの受け皿として機能している
- **AIコーディングの 2 系統**  
  「CLI 本体」と「ワークフロー/プロンプト補助」を分ける考え方自体は有効

## 問題点

### 1. ラベル揺れが残っている

- `AIコーディング (ワークフロー・プロンプト・開発補助ツール)`
- `AIコーディング (プロンプト・開発補助ツール)`

この 2 つは実質同義です。UI 側では吸収できていても、データ上で残ると集計・検索・今後の運用判断がぶれます。

### 2. AI エージェント系カテゴリが細かく割れすぎている

以下 3 つは、現時点の件数では独立維持のメリットが小さいです。

- `AIエージェント (自律基盤・特化アプリ)` 25 件
- `AIエージェント (基盤・コンポーネント)` 2 件
- `AIエージェント・マルチエージェント・オーケストレーション` 1 件

利用者視点では「エージェント系」で一括して見たい可能性が高く、編集側にも判断コストを増やしています。

### 3. `汎用アプリ・AI基盤・IoT` が広すぎる

このカテゴリには、性質の異なるものが混在しています。

- AI / データ基盤: `google-research/timesfm`
- BI / 可視化基盤: `apache/superset`
- 業務アプリ: `twentyhq/twenty`

件数は 24 件と中規模ですが、**カテゴリ名が広すぎるため何を入れる場所か直感的に分かりにくい**状態です。

### 4. メディアカテゴリにエージェント/自動化系の外れ値が入っている

`メディア作成・Web生成 (動画・画像・UI)` は本来有効なカテゴリですが、次のような外れ値が混じっています。

- `browser-use/browser-use`
- `Visual-Agent/DeepEyesV2`
- `badlogic/pi-mono`

これらは「メディア生成」より、**AI エージェント / ブラウザ操作 / 自律実行**として捉える方が自然です。

### 5. 同一 repo の分類ぶれが一部で発生している

複数回記事化された repo のうち、少なくとも次のケースでジャンルぶれがありました。

| repo | 現行の揺れ | 推奨 |
| --- | --- | --- |
| `SHANRAISSHAN/CLAUDE-CODE-BEST-PRACTICE` | `CLI本体・拡張機能` / `ワークフロー・プロンプト・開発補助ツール` | 後者に統一 |
| `PASCALORG/EDITOR` | `メディア作成・Web生成` / `AIコーディング (プロンプト・開発補助ツール)` | 前者に統一 |

件数としては多くありませんが、**編集基準の曖昧さ**を示しています。

## 提案するカテゴリ体系

### 基本方針

- 新カテゴリを増やしすぎない
- 既存の利用価値が高いカテゴリは維持する
- 少数派カテゴリは統合する
- 分かりにくいカテゴリ名だけを最小限リネームする

## 運用ルール提案

今後の分類ぶれを減らすため、以下の 3 ルールを推奨します。

1. **実装形態ではなく、利用者が得る主価値で分類する**  
   例: CLI で配布されていても、価値の中心が音声処理ならメディア系
2. **エージェントがブラウザやツールを操作する repo は、まず AI エージェントを優先候補にする**
3. **同一 repo の過去記事がある場合は、そのカテゴリを先に参照してから新規分類する**

## 実施優先度の提案

実作業を行う場合は、次の順で十分です。

1. 正規カテゴリ名を 9 個に固定する
2. ラベル揺れを一括統合する
3. 外れ値 repo のみ個別に見直す

この順なら、**大きな構造変更なしで分類精度と見通しを改善**できます。

## 最終判断

GitHub Watcher のカテゴリ体系は、**ゼロから作り直す必要はありません**。  
一方で、現状のままでは「AI エージェント周りの分裂」「広すぎる汎用カテゴリ」「一部の外れ値混入」により、今後の記事追加でぶれが拡大しやすい状態です。

したがって最適解は、**カテゴリ追加よりも、正規ラベルを 9 個に整理し、外れ値を戻し、分類ルールを明文化すること**です。
