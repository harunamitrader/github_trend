Claude Code を劇的に改善する Karpathy 流 CLAUDE.md 実践ガイド

推測を排し、シンプルさを貫く。AI 研究者 Andrej Karpathy 氏が提唱する「LLM コーポレート・ガイドライン」を日本語で徹底解説。

作成日 2026-04-17
カテゴリ AI 開発手法
対象 Claude Code / AI Agents

はじめに

AI 研究・開発の第一人者である Andrej Karpathy 氏が指摘した、「LLM がコーディング中に陥りがちな落とし穴」を回避するための具体的な行動指針を解説します。Claude Code をはじめとする AI エージェントの挙動を、よりシンプルで、確実で、余計なことをしない「熟練エンジニア」へと近づけるための高品質な CLAUDE.md の作成を目指します。

なぜこれが必要か（LLM の弱点）

Karpathy 氏は、現在の LLM コーディングには以下の 3 つの大きな問題があるとしています。

  • 勝手な推測： 本来質問すべき不明点に対し、勝手に憶測で進めてしまう。
  • 複雑化（Bloat）： 100 行で済むものを 1000 行の過剰な抽象化で構築してしまう。
  • 無駄な変更： 関係ないコメントを消したり、壊れていない隣接コードを「修正」してしまう。

解決のための 4 つの原則

Karpathy 氏の知見を元に提案されている、4 つの行動ルールを深掘りします。

1. 実装前にまず思考せよ (Think Before Coding)

勝手に推測せず、曖昧な点はまず質問することを優先します。複数の解釈がある場合は独断せず、ユーザーに選択肢を提示させます。これにより、「作ってからやり直し」というコストを最小化します。

2. シンプルさを最優先に (Simplicity First)

「将来のため」といった憶測での機能追加や、不必要な抽象化を禁じます。目的を達成する最小限のコードにこだわります。シニアエンジニアがコードを見て「過剰だ」と感じるものはすべて排除します。

3. 「外科手術」のようなピンポイント変更 (Surgical Changes)

修正が必要な箇所だけを触り、関係ないリファクタリングやフォーマット変更を行わないよう徹底します。差分（Diff）を極限までクリーンに保つことが、信頼性に繋がります。

4. ゴール駆動の実行 (Goal-Driven Execution)

「とりあえず修正して」を、「テストを成功させる」などの検証可能なゴールに変換させます。特に多段階のタスクでは、実行ごとに検証を挟むループを回します。具体的な成功指標を持たせることが重要です。

CLAUDE.md 日本語記述例（テンプレート）

自身のプロジェクトに導入する際、以下の内容を CLAUDE.md に記述してください。Claude Code のシステムプロンプトとして機能し、挙動が劇的に安定します。

# Behavior Guidelines

AI特有のコーディングミスを減らすため、以下の原則を厳守してください。

## 1. 実装前の思考と確認
- 勝手な推測をせず、不明点があれば実装前に必ず質問すること。
- 複数の解釈ができる場合は、独断で選ばず選択肢を提示すること。
- よりシンプルな設計が可能なら、それを提案すること。

## 2. シンプルさの追求
- 要求されていない機能や「将来のための拡張性」を追加しない。
- 1回しか使わないコードを抽象化（共通化）しない。
- 200行のコードが50行で書けるなら、迷わず書き直すこと。

## 3. 外科手術的な変更
- 依頼された修正に関係のない箇所（隣接コード、コメント、フォーマット）を触らない。
- 元からあるデッドコードを見つけた場合は、削除せず報告のみ行う。
- 既存のコードスタイルに100%合わせること。

## 4. ゴールの定義と検証
- タスクを「検証可能なゴール」に変換してから着手すること。
- (例: 「バグ修正」→「失敗するテストを書き、それをパスさせる」)
- 計画を「[手順] → [検証]」の形式で立ててから実行すること。

まとめ

AI エージェントは非常に強力ですが、デフォルトでは「頑張りすぎてしまう（過剰構築）」という癖があります。Andrej Karpathy 氏流の厳格な節制ルールを CLAUDE.md で定義することで、AI はあなたの意図を汲み取り、より洗練されたコードを生成するようになるでしょう。
