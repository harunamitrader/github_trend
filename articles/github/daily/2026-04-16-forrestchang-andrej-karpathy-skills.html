これは何か

AI 研究・開発の第一人者である Andrej Karpathy 氏が指摘した、「LLM がコーディング中に陥りがちな落とし穴」を回避するための具体的な行動指針をまとめたリポジトリです。Claude Code をはじめとする AI エージェントの挙動を、よりシンプルで、確実で、余計なことをしない「熟練エンジニア」へと近づけるための高品質な CLAUDE.md テンプレートを提供しています。

なぜこれが必要か（LLM の弱点）

Karpathy 氏は、現在の LLM コーディングには以下の 3 つの大きな問題があるとしています。

  • 勝手な推測： 本来質問すべき不明点に対し、勝手に憶測で進めてしまう。
  • 複雑化（Bloat）： 100 行で済むものを 1000 行の過剰な抽象化で構築してしまう。
  • 無駄な変更： 関係ないコメントを消したり、壊れていない隣接コードを「修正」してしまう。

解決のための 4 つの原則

このリポジトリでは、これらの弱点を克服するために以下の 4 つのルールを CLAUDE.md に含めることを推奨しています。

1. 実装前にまず思考せよ (Think Before Coding)

勝手に推測せず、曖昧な点はまず質問することを優先します。複数の解釈がある場合は独断せず、ユーザーに選択肢を提示させます。

2. シンプルさを最優先に (Simplicity First)

「将来のため」といった憶測での機能追加や、不必要な抽象化を禁じます。目的を達成する最小限のコードにこだわります。

3. 「外科手術」のようなピンポイント変更 (Surgical Changes)

修正が必要な箇所だけを触り、関係ないリファクタリングやフォーマット変更を行わないよう徹底します。差分（Diff）を極限までクリーンに保ちます。

4. ゴール駆動の実行 (Goal-Driven Execution)

「とりあえず修正して」を、「テストを成功させる」などの検証可能なゴールに変換させます。特に多段階のタスクでは、実行ごとに検証を挟むループを回します。

推奨：CLAUDE.md 日本語記述例

自身のプロジェクトに導入する際、以下の内容を CLAUDE.md に追記（または上書き）することで、AI のコーディング精度が大幅に向上します。

## 行動指針 (Karpathy-style)

AI特有のコーディングミスを減らすため、以下の原則を厳守してください。

### 1. 実装前の思考と確認
- 勝手な推測をせず、不明点があれば実装前に必ず質問すること。
- 複数の解釈ができる場合は、独断で選ばず選択肢を提示すること。
- よりシンプルな設計が可能なら、それを提案すること。

### 2. シンプルさの追求
- 要求されていない機能や「将来のための拡張性」を追加しない。
- 1回しか使わないコードを抽象化（共通化）しない。
- 200行のコードが50行で書けるなら、迷わず書き直すこと。

### 3. 外科手術的な変更
- 依頼された修正に関係のない箇所（隣接コード、コメント、フォーマット）を触らない。
- 元からあるデッドコードを見つけた場合は、削除せず報告のみ行う。
- 既存のコードスタイルに100%合わせること。

### 4. ゴールの定義と検証
- タスクを「検証可能なゴール」に変換してから着手すること。
- (例: 「バグ修正」→「失敗するテストを書き、それをパスさせる」)
- 複雑なタスクでは [手順] → [検証] の形式で計画を立て、ループを回すこと。
